2. Data Architecture — Four Structures

Each pillar uses a different data structure optimized for its access pattern.

🧬 Codebase Snapshots Ring Buffer · 30 slots

Sampled on every build. Last 30 daily snapshots.
DSA: Ring Buffer — overwrites oldest. O(1) trend read.

ColTypeRole
repo_idTEXTTenant
slotINTRing index 0–29
sampled_atTIMESTAMPTZWhen build ran
ts_errorsINTnpx tsc --noEmit count
ts_error_freeBOOLClean build?
files_totalINTSource files
lines_totalINTTotal lines
build_duration_msINTCompile time
dependenciesINTnpm ls count
outdated_depsINTnpm outdated count
🌐 API Health Sliding Window · 24h

Middleware logs every API call. Window aggregates last 24h.
DSA: Sliding Window — O(1) per request, O(W) aggregate.

ColTypeRole
repo_idTEXTTenant
endpointTEXT/api/agent-ops
status_codeINT200/400/500
duration_msINTLatency
error_messageTEXTStack trace if 500
created_atTIMESTAMPTZ↓ Clustered
Window: SELECT COUNT(*) WHERE created_at > now() - 24h.
Aggregate by status_code GROUP BY hourly for trend chart.
🐛 Issues Priority Queue

Ordered by severity × age. Critical issues bubble to top.
DSA: Priority Queue — ORDER BY severity DESC, age DESC.

ColTypeRole
repo_idTEXTTenant
severityINT1=critical 2=high 3=medium 4=low
titleTEXTIssue title
sourceTEXTtsc / build / api / manual
resolvedBOOLClosed?
resolution_time_hFLOATTime to close
opened_atTIMESTAMPTZCreated
resolved_atTIMESTAMPTZClosed
Query: ORDER BY severity ASC, opened_at DESC WHERE resolved=false.
Avg resolution: AVG(resolution_time_h) WHERE resolved=true (last 30d).
🧠 TOON Quality Composite Index

Weighted score from 4 metrics. Normalized to 0–100.
DSA: Weighted Composite — 4 inputs → 1 health score.

MetricWeightSource
Compression %40%Engine stats
Bundle size25%.toon/ size
Corpus freshness20%Last compile age
Index coverage15%Chunks / files ratio
score = (comp% × 0.4) + (size_score × 0.25) + (fresh_score × 0.2) + (cov_score × 0.15)

3. Schema — 3 New Tables

Added to the existing v2 schema. All repo-scoped.

-- Codebase snapshots (ring buffer, 30 daily slots per repo) CREATE TABLE toongine_codebase_snapshots ( repo_id TEXT NOT NULL, slot INT NOT NULL, -- 0–29, overwrites oldest sampled_at TIMESTAMPTZ DEFAULT now(), ts_errors INT DEFAULT 0, ts_error_free BOOLEAN DEFAULT true, files_total INT DEFAULT 0, lines_total INT DEFAULT 0, build_duration_ms INT DEFAULT 0, dependencies INT DEFAULT 0, outdated_deps INT DEFAULT 0 ); -- API health log (append-only, sliding window query) CREATE TABLE toongine_api_health ( id BIGSERIAL PRIMARY KEY, repo_id TEXT NOT NULL, endpoint TEXT NOT NULL, status_code INT NOT NULL, duration_ms INT DEFAULT 0, error_message TEXT DEFAULT '', created_at TIMESTAMPTZ DEFAULT now() ); -- Issues tracker (priority queue) CREATE TABLE toongine_issues ( id BIGSERIAL PRIMARY KEY, repo_id TEXT NOT NULL, severity INT NOT NULL, -- 1=critical,2=high,3=medium,4=low source TEXT DEFAULT 'manual', -- tsc|build|api|manual title TEXT NOT NULL, detail TEXT DEFAULT '', resolved BOOLEAN DEFAULT false, resolution_time_h FLOAT, opened_at TIMESTAMPTZ DEFAULT now(), resolved_at TIMESTAMPTZ ); -- Indexes CREATE INDEX idx_codebase_repo ON toongine_codebase_snapshots(repo_id, slot); CREATE INDEX idx_api_repo_time ON toongine_api_health(repo_id, created_at DESC); CREATE INDEX idx_issues_open ON toongine_issues(repo_id, severity) WHERE resolved=false; -- RLS ALTER TABLE toongine_codebase_snapshots ENABLE ROW LEVEL SECURITY; ALTER TABLE toongine_api_health ENABLE ROW LEVEL SECURITY; ALTER TABLE toongine_issues ENABLE ROW LEVEL SECURITY; CREATE POLICY "anon read" ON toongine_codebase_snapshots FOR SELECT TO anon USING(true); CREATE POLICY "anon read" ON toongine_api_health FOR SELECT TO anon USING(true); CREATE POLICY "anon read" ON toongine_issues FOR SELECT TO anon USING(true);

4. Data Collection Pipeline

How each pillar gets data — automated where possible, manual entry for issues.

🧬 Codebase Sampler

Trigger: Post-build hook OR daily cron
Input: npx tsc --noEmit output, file count
Output: UPSERT toongine_codebase_snapshots[slot]

# scripts/sample-codebase.py ts_errors=$(npx tsc --noEmit 2>&1 | grep "error TS" | wc -l) files=$(find src -name "*.ts" -o -name "*.tsx" | wc -l) # → POST to Supabase
🌐 API Logger Middleware

Trigger: Every API request (Next.js middleware)
Input: Request URL, status code, duration
Output: INSERT toongine_api_health

// middleware.ts export function middleware(req) { const start = Date.now() const res = NextResponse.next() res.headers.set('X-Health-Log', 'true') // After response: POST to Supabase }
🧠 TOON Quality Snapshot

Trigger: On every toongine compile
Input: Compression ratio, bundle size, chunks
Output: Reuses toongine_snapshots + health score

# Already tracked in toongine_snapshots.efficiency_pct # Just query + compute the composite score
🐛 Issues (Manual + Auto)

Auto: TypeScript errors → issues table
Manual: Via dashboard "Add Issue" button
Output: INSERT toongine_issues

# Auto: if ts_errors > 0 → create issue INSERT INTO toongine_issues(severity,title,source) VALUES (2, 'TypeScript errors found', 'tsc')

7. Health Score Formula

Weighted composite index. Each pillar normalized to 0–100 before weighting.

PillarWeightHow It's Calculated
🧬 Codebase30% 100 if ts_errors=0, -10 per error (floor 0). -5 per outdated dep (cap -20).
🌐 API30% SUCCESS_RATE_24H × 100. <95% = yellow, <90% = red.
🧠 TOON25% efficiency_pct from snapshots. Already computed by pipeline.
🐛 Issues15% 100 if 0 critical, -15 per critical, -5 per high, -2 per medium. Floor 0.
health_score = codebase × 0.30 + api × 0.30 + toon × 0.25 + issues × 0.15 // Color thresholds score ≥ 80 → 🟢 Healthy score ≥ 50 → 🟡 Degraded score < 50 → 🔴 Critical
