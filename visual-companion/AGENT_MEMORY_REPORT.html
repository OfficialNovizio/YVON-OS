Critical Finding
The agent memory system has never been used. Every MEMORY.md file across all 17 agents is blank — they contain only the template scaffold with dashes. Zero knowledge has been persisted to any agent's long-term memory. The system was designed but never activated.

Why Memory Was Never Populated

01

Wrong Execution Context

The memory write-back instructions ("After every task, update your MEMORY.md") lived in the agent system prompts. System prompts run inside web API calls — the Anthropic API endpoint. Claude inside an API call cannot access the file system. The instructions were physically impossible to execute.

02

File Access Requires Claude Code

MEMORY.md files can only be written when Claude has the Edit/Write tools available — that only happens in a Claude Code session. API calls (via /api/claude, /api/team-chat) have no file-write capability. The two contexts were conflated.

03

Instructions Were Stripped (Token Fix)

In the recent token optimization pass, the "After every task, update MEMORY.md" block was correctly stripped from all system prompts (it was dead weight — ~100 tokens per prompt). Those instructions are now gone, removing even the theoretical write path.

04

No Claude Code Trigger Was Defined

The memory files exist in /departments/ but there was never a CLAUDE.md instruction telling Claude Code which memory to read or when to write to it. Without a trigger in CLAUDE.md, Claude Code never knew to consult these files.

What the Memory System Was Designed to Do
Each agent's MEMORY.md was meant to hold a growing log of: completed tasks and their outcomes, decisions made and reasoning, key learnings, open threads, and cross-agent context. The idea was that when a Claude Code session started to work with (e.g.) Dev, it would first read departments/technical/dev-lead/MEMORY.md to "catch up" on what Dev had last worked on — giving continuity across sessions. Good design. Broken implementation path.

    CLAUDE.md update

    Architecture rule
